# MolVis 0.0.2 Pre-Release Fix List

Action plan for releasing `molvis` `0.0.2` across:

- `core`
- `page`
- `vsc-ext`
- `python` widget

This document is derived from the pre-release review and is ordered by release risk.

## Release Verdict

Current state: `Not ready for release`

The release should not ship until all blocker items are closed and re-verified.

## Blockers

### 1. Unify version, license, and release scope

Problem:

- Package versions are inconsistent across the monorepo.
- Root documentation still references older versions.
- Python package license does not match the rest of the repository.
- There is no single release note defining what `0.0.2` contains.

Files to align:

- `package.json`
- `core/package.json`
- `page/package.json`
- `python/package.json`
- `python/pyproject.toml`
- `python/src/molvis/__init__.py`
- `vsc-ext/package.json`
- `README.md`
- `vsc-ext/CHANGELOG.md`

Done when:

- All four deliverables use an intentional and documented version strategy.
- License metadata is consistent or explicitly justified.
- README badges and changelog entries match the release target.
- A release note exists for the exact candidate being shipped.

### 2. Fix all broken quick-start and API examples

Problem:

- Current TypeScript and Python examples do not match the real public API.
- `readFrame(...)` examples are inconsistent with the implementation signature.

Files to fix:

- `README.md`
- `docs/getting-started.md`
- `core/README.md`
- `python/README.md`
- `docs/api/core.md`

Done when:

- Every documented quick-start example uses the current public API.
- The examples are syntax-checked manually against the actual exports.
- `readFrame(content, filename)` is documented consistently everywhere.

### 3. Repair the VS Code extension test lane

Problem:

- `npm run test:unit -w vsc-ext` fails at compile time with `TS5096`.
- CI does not currently enforce `vsc-ext` tests.

Files to fix:

- `vsc-ext/tsconfig.test.json`
- `vsc-ext/package.json`
- `.github/workflows/ci.yml`
- optionally root `package.json`

Recommended acceptance checks:

```bash
npm run test:unit -w vsc-ext
npm run test:integration -w vsc-ext
```

Done when:

- The extension test compile step succeeds.
- Unit tests run instead of failing in TypeScript configuration.
- CI includes the extension test lane.

## High Priority

### 4. Add explicit release gates for `page` and `python`

Problem:

- `page` has no test script.
- `python` has no discovered test suite and no verified package build in a clean environment.

Files to update:

- `page/package.json`
- `python/package.json`
- `python/pyproject.toml`
- root `package.json`
- `.github/workflows/ci.yml`

Minimum acceptable gate:

- `page`: add at least one smoke test or component-level sanity check.
- `python`: add at least one import/build smoke test and one minimal widget command-path test.

Recommended acceptance checks:

```bash
npm run build -w page
npm run test -w page
python3 -m pytest python
python3 -m build python
```

### 5. Make Python widget packaging reproducible

Problem:

- The widget runtime requires `src/molvis/dist/index.js`.
- Current packaging docs are stale and the build flow is not clearly tied to wheel/sdist creation.

Files to review:

- `python/README.md`
- `python/pyproject.toml`
- `python/package.json`
- `python/src/molvis/scene.py`

Done when:

- Packaging docs reference the real `python/` package path.
- Frontend asset generation is a documented precondition for publishing.
- Wheel/sdist creation is verified from a clean environment.

### 6. Add release automation coverage for all deliverables

Problem:

- Root `build:all` does not include `vsc-ext`.
- Root `test` only covers `core`.
- Current release docs only cover `core`.

Files to review:

- `package.json`
- `docs/index.md`
- `docs/release/core-release-checklist.md`

Done when:

- Root-level scripts reflect the actual release scope.
- Release docs cover `core`, `page`, `vsc-ext`, and `python`.

## Medium Priority

### 7. Fix confirmed Python widget API defect

Problem:

- `Molvis.get_frontend_instance_count()` does not wait for an RPC response and effectively returns `0` in normal operation.

Files to fix:

- `python/src/molvis/scene.py`

Done when:

- The method requests a response explicitly.
- The method is covered by a regression test.

### 8. Refresh stale package docs and structure descriptions

Problem:

- `page` README still describes a minimal canvas demo.
- `vsc-ext` README still documents removed structure.
- Root docs mention paths and components that do not reflect the current repo state.

Files to update:

- `page/README.md`
- `vsc-ext/README.md`
- `README.md`

Done when:

- Package READMEs describe the current product shape.
- Directory diagrams match real files.
- Development and packaging commands are current.

### 9. Improve source-driven API documentation coverage

Problem:

- Only `core` is wired into TypeDoc.
- Public APIs in `page`, `python`, and `vsc-ext` are documented manually or not at all.
- Some `core` exports still rely on hand-maintained docs.

Files to review:

- `typedoc.json`
- `core/typedoc.json`
- `core/src/index.ts`
- `python/src/molvis/scene.py`
- `vsc-ext/package.json`

Done when:

- Public APIs have stable source-level documentation.
- Generated or source-bound docs cover the APIs users are expected to call.

## Verification Matrix

Run this full gate before tagging the release candidate:

```bash
npm run build:all
npm run build -w vsc-ext
npm run test -w core
npm run test -w vsc-ext
npm run test -w page
python3 -m pytest python
python3 -m build python
npm_config_cache=/tmp/npm-cache npm pack --dry-run -w core
```

## Suggested Execution Order

1. Decide and apply the version strategy for all four deliverables.
2. Fix README/API docs so public examples match the real APIs.
3. Repair `vsc-ext` tests and add them to CI.
4. Add minimal smoke tests for `page` and `python`.
5. Verify clean Python packaging.
6. Expand release runbooks from `core` only to all release targets.

## Next Slice

The safest next implementation slice is:

1. Repair `vsc-ext` test configuration and CI coverage.
2. Fix the broken public docs examples.
3. Then decide whether to normalize versions to `0.0.2` or cut a different target version.
